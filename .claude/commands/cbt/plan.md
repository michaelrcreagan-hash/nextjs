<objective>
Create a detailed, step-by-step BUILD_PLAN.md that maps out exactly how the strategy code will be built.
Each step is atomic, testable, and has clear inputs/outputs. The plan adapts to strategy complexity and engine choice.
</objective>

<execution_context>
@strategies/{active}/DISCOVERY.md
@strategies/{active}/RESEARCH.md (if exists)
@strategies/{active}/EDA.md (if exists)
@strategies/{active}/config.yaml
@strategies/{active}/.cbt/state.yaml
@~/.claude/cbt-framework/references/strategy-types.md
</execution_context>

<principles>
- Every step must be atomic and independently testable
- Dependencies between steps must be explicit
- The plan should be specific to THIS strategy, not generic
- Engine choice (pandas vs fast) determines implementation details
- Complexity level determines number and type of steps
</principles>

<process>

## 1. Load Context

Read all available documents:
- DISCOVERY.md → strategy logic, entry/exit rules, data requirements
- RESEARCH.md → implementation patterns, known pitfalls
- EDA.md → data characteristics, feature insights (if exists)
- config.yaml → backtest parameters
- state.yaml → engine choice, project_type, mode

## 2. Determine Plan Template

Based on complexity + engine + project_type:

### Simple Strategy (indicator-based) + pandas
1. Data Pipeline (src/data_loader.py)
2. Signal Generation (src/signals.py)
3. Strategy Integration (strategy.py)
4. Backtest Runner (backtest.py)
5. Baseline Run

### Simple Strategy + fast engine
1. Data Pipeline - Polars (src/data_loader.py)
2. Signal Generation - NumPy (src/signals.py)
3. Strategy Integration - Numba (strategy.py)
4. Backtest Runner - Numba JIT (backtest.py)
5. Baseline Run

### Medium Strategy (feature-based) + pandas
1. Data Pipeline (src/data_loader.py)
2. Feature Engineering (src/features.py)
3. Signal Generation (src/signals.py)
4. Strategy Integration (strategy.py)
5. Backtest Runner (backtest.py)
6. Baseline Run

### Medium Strategy + fast engine
1. Data Pipeline - Polars (src/data_loader.py)
2. Feature Engineering - NumPy arrays (src/features.py)
3. Signal Generation - NumPy (src/signals.py)
4. Strategy to NumPy arrays (strategy.py)
5. Backtest Runner - Numba JIT (backtest.py)
6. Baseline Run

### Complex Strategy (ML pipeline) + pandas
1. Data Pipeline (src/data_loader.py)
2. Feature Engineering (src/features.py)
3. Label Generation (src/labeling.py)
4. Model Training (src/model.py)
5. Signal Generation (src/signals.py)
6. Strategy Integration (strategy.py)
7. Backtest Runner (backtest.py)
8. Walk-Forward Validation
9. Baseline Run

### Complex Strategy (ML) + fast engine
1. Data Pipeline - Polars (src/data_loader.py)
2. Feature Engineering - NumPy (src/features.py)
3. Label Generation - NumPy (src/labeling.py)
4. Model Training (src/model.py)
5. Signal Generation - NumPy (src/signals.py)
6. Strategy to NumPy arrays (strategy.py)
7. Backtest Runner - Numba JIT (backtest.py)
8. Walk-Forward Validation
9. Baseline Run

## 3. Generate BUILD_PLAN.md

```markdown
# Build Plan: {strategy_name}

**Date:** {date}
**Complexity:** {Simple / Medium / Complex}
**Engine:** {pandas / fast}
**Project Type:** {indicator / ml / hybrid}
**Estimated Steps:** {N}

---

## Overview

{2-3 sentences describing what will be built and the approach}

## Dependencies

```
Step 1 ─→ Step 2 ─→ Step 3 ─→ ...
                  └─→ Step N (parallel if possible)
```

---

## Step 1: {Step Name}

**File:** `{output_file}`
**Depends on:** None
**Complexity:** {Low / Medium / High}
**Checkpoint:** {Yes/No - whether to save intermediate results}

### What it does
{Description of what this step builds}

### Key implementation details
- {Detail 1 from DISCOVERY.md / RESEARCH.md}
- {Detail 2}
- {Any gotchas from research}

### Inputs
- {What data/files this step needs}

### Outputs
- {What files/artifacts this step produces}

### Verification
- {How to verify this step works correctly}
- {Specific test or validation to run}

---

## Step 2: {Step Name}
...

---

## Final Checklist

Before running baseline:
- [ ] All source files created and import correctly
- [ ] No lookahead bias in features (all use .shift(1))
- [ ] Config.yaml parameters wired correctly
- [ ] Data loads without errors
- [ ] Signals generate for full dataset
{if fast engine:}
- [ ] Polars lazy frames evaluated correctly
- [ ] NumPy array shapes consistent
- [ ] Numba functions compile without errors

---

*Generated by CBT Framework /cbt:plan*
```

## 4. Update State

```yaml
phase: build
phases_completed:
  plan: true
build:
  plan: [{step names list}]
  current_step: null
  progress: "0/{N}"
```

## 5. Output Summary

```
Build Plan created!

Strategy: {name}
Complexity: {level}
Engine: {pandas/fast}
Steps: {N}

Review BUILD_PLAN.md then run /cbt:build to start building.

Tip: Each step will be implemented and tested before moving to the next.
{If YOLO mode: "YOLO mode: steps will auto-approve."}
```

</process>

<constraints>
- Plan MUST be specific to the strategy, not a generic template
- Each step MUST have clear verification criteria
- Dependencies between steps MUST be explicit
- DO reference specific findings from RESEARCH.md and EDA.md when relevant
- DO include engine-specific implementation notes
- DO NOT write any code - only the plan
</constraints>

<success_criteria>
- [ ] BUILD_PLAN.md created with all steps
- [ ] Each step has: file, dependencies, complexity, verification
- [ ] Plan adapts to engine choice (pandas vs fast)
- [ ] Plan adapts to project type (indicator vs ML vs hybrid)
- [ ] Research/EDA findings incorporated into step details
- [ ] State updated to build phase
</success_criteria>
